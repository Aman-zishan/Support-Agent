Foundations

What is an AI Agent?

The Agent Loop

An agent combines an LLM with tools, memory, and a decision loop. It observes input, reasons about what to do, takes action by calling tools, then observes the result and decides if more work is needed.

Observe Reason Act Repeat
Agent vs Chatbot vs Workflow

A chatbot responds to messages. A workflow executes predefined steps. An agent decides which tools to call and when, iterating until the task is complete.

Use agents when the path to completion is not predetermined.

// The agent loop (conceptual)
while (!task.isComplete()) {
  const observation = await observe(input);
  const plan = await llm.reason(observation);
  const result = await tools[plan.tool].execute(
    plan.args
  );
  input = result;
}
Key Insight
Agents are autonomous decision-makers. They choose which tool to call and when to stop. This flexibility is powerful but requires safety guardrails.

Security Primitives

Agent Primitives Part 1: Safety First

🛡
Least Authority

Give agents the minimum permissions they need. A billing agent should look up orders but never delete accounts.

agents/triage.ts
// Triage agent: ZERO tools
// It only classifies tickets
export const triageAgent = new Agent({
  model: groq("llama-3.3-70b-versatile"),
  // No tools property!
});

agents/billing.ts
// Billing: can lookup, can NOT refund
tools: {
  lookupCustomer, getOrderHistory
  // No processRefund!
}
🔄
Idempotency & Reversibility

Every tool call must be safe to retry. Prefer reversible actions: draft → review → send, not send directly. Soft-delete over hard-delete.

✓ Good: agent.recommend(refund) → human.approve() → system.process()
✗ Bad: agent.processRefund() directly
✓ Good: Soft-delete with recovery window
✗ Bad: Hard-delete with no undo

The Trust Spectrum & HITL

Auto ≤$50

⚡
Full Automation

Low-risk repetitive tasks like ticket classification or password reset instructions can be fully automated. The agent handles these end-to-end without pausing.

👤
Human Approval

High-risk actions like refunds, account deletions, or external communications require human sign-off. Mastra uses suspend() to pause and resume() to continue.

Code Deep Dive

HITL: Suspend & Resume in Action

workflows/support-workflow.ts:335-408

execute: async ({ inputData, resumeData, suspend }) => {
  const { action, refundAmount, orderId } = inputData;

  // Not a refund? Pass through
  if (action !== "refund") return { ... };

  // Auto-approve small refunds (≤ $50)
   if (refundAmount <= 50) {
     await processRefund({ orderId, amount, reason });
     return { action: "refund-auto-approved" };
   }

  // Large refund → SUSPEND for human
  if (!resumeData) {
    await suspend({
      message: `Refund of $${refundAmount}
        requires manager approval`,
      refundAmount, orderId, customerId,
      agentRecommendation: reason,
    });
  }

  // Resumed with manager decision
  const { approved, managerNote } = resumeData;
  if (approved) await processRefund({ ... });
}
How It Works
1Workflow runs normally
2suspend() pauses execution
3State persisted to database
4Manager reviews & decides
5resume() continues execution
Suspend Schema
{ message, refundAmount,
  orderId, customerId,
  agentRecommendation }

{ approved: boolean,

Defense in Depth

Guardrails for AI Agents

🛑
Input Validation

Validate input length, format, and content before any agent sees it. Use Mastra's PromptInjectionDetector for LLM-based detection instead of fragile regex patterns.

📈
Output Checks

Parse agent responses with Zod for schema conformance. Use PIIDetector to automatically redact sensitive data. Apply rate limits on tool calls.

Mastra Built-in Processors

import {
  PromptInjectionDetector,
  PIIDetector
} from "@mastra/core/processors";

// LLM-based injection detection
const injectionDetector = new PromptInjectionDetector({
  model: groq("llama-3.3-70b-versatile"),
  detectionTypes: ["injection",
    "jailbreak", "system-override"],
  threshold: 0.8,
  strategy: "block",
});

// LLM-based PII detection + redaction
const piiDetector = new PIIDetector({
  model: groq("llama-3.3-70b-versatile"),
  detectionTypes: ["ssn", "credit-card"],
  strategy: "redact",
  redactionMethod: "mask",
});

Scalability Deep Dive

Background Workflows & Workers

⚙
Suspend = Free Resources

When a workflow calls suspend(), it serializes state to the database and releases the worker. No threads blocked waiting for human approval. The worker can process other requests.

📄
Step-Based Processing

Each workflow step is a discrete processing unit. Steps can run on different workers. State passes between steps via validated schemas.

💬
Thread-Based Concurrency

Each conversation thread has isolated memory. Multiple threads run simultaneously. Sliding window of 40 messages keeps context efficient.

Scalability Architecture

// Workflow steps as processing units
export const supportWorkflow = createWorkflow({
  id: "customer-support-workflow",
})
  .then(validateTicketStep)  // Worker 1
  .then(triageStep)         // Worker 2
  .then(specialistStep)     // Worker 3
  .then(refundApprovalStep)  // Suspends!
  .commit();

// Memory: Per-thread isolation
memory: new Memory({
  options: {
    lastMessages: 40,
    workingMemory: {
      enabled: true,
      template: `<customer_context>
        <customerId/>
        <name/> <plan/>
      </customer_context>`
    }
  }
})
Production tip: Replace file:./support.db with Turso (managed LibSQL) or Postgres for multi-instance deployments.

Building Tools for Specialist Agents

tools/index.ts - lookupCustomer

export const lookupCustomerTool = createTool({
  id: "lookup-customer",
  inputSchema: z.object({
    customerId: z.string()
  }),
  outputSchema: z.object({
    name: z.string(),
    email: z.string(),
    plan: z.string(),
    found: z.boolean(),
  }),
  execute: async ({ context }) => {
    const customers = {
      "C001": { name: "Alice",
        plan: "pro" },
      "C002": { name: "Bob",
        plan: "starter" },
    };
    return customers[context.customerId];
  }
});
tools/index.ts - processRefund

// NOT a tool - a plain function
// Agents cannot call this directly!
export async function processRefund(input: {
  orderId: string;
  amount: number;
  reason: string;
}) {
  return {
    success: true,
    refundId: `REF-${Date.now()}`,
    message: "Refund processed",
  };
}
Key Security Pattern

processRefund is a plain function, NOT a Mastra tool. No agent can call it directly. It's only invoked from the workflow's refund-approval step, after human approval for amounts > $50.

In production: Replace mock data with real database queries and Stripe API calls. The Zod schemas stay the same.
